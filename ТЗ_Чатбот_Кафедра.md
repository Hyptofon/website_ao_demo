## 1. Загальні положення

### 1.1. Мета проєкту

Метою проєкту є розробка інтелектуального веб-чат-бота для кафедри університету, що забезпечує автоматизовані відповіді на запитання абітурієнтів у режимі 24/7. Система вирішує проблему надмірного навантаження на адміністративний персонал кафедри через величезний потік однотипних запитань про вступ, документи, розклад та спеціальності. Чат-бот ґрунтується на технології RAG (Retrieval-Augmented Generation) — він не генерує відповіді «з голови», а витягує актуальну інформацію виключно з офіційних документів кафедри, що гарантує точність і достовірність наданих відповідей.

### 1.2. Об'єкт та предмет проєкту

**Об'єкт проєкту** — кафедра університету та абітурієнти, які зацікавлені у вступі на конкретну спеціальність.

**Предмет проєкту** — процеси надання інформаційної підтримки абітурієнтам: відповіді на запитання про вступ, правила прийому, необхідні документи, розклад, навчальні плани та спеціальності кафедри.

**Опис взаємодії:** система надає абітурієнтам зручний інтерфейс для отримання миттєвих відповідей на основі офіційних документів кафедри. Адміністратор кафедри самостійно підтримує актуальність бази знань через зрозумілу адмін-панель — без залучення розробника.

### 1.3. Основні терміни та визначення

| Термін | Визначення |
|---|---|
| **RAG** | Retrieval-Augmented Generation — технологія, що поєднує пошук у базі знань з генерацією відповіді мовною моделлю |
| **LLM** | Large Language Model — велика мовна модель (у даному проєкті: Google Gemini) |
| **Embedding** | Векторне представлення тексту для семантичного пошуку |
| **Чанк** | Фрагмент документа фіксованого розміру, що зберігається у векторній базі |
| **Vector DB** | Векторна база даних для зберігання та пошуку embedding-ів (Qdrant) |
| **Streaming** | Потокова передача відповіді токен за токеном, як у ChatGPT |
| **Rate Limiting** | Обмеження кількості запитів від одного користувача за одиницю часу |
| **Off-topic** | Запит, що не стосується тематики кафедри/вступу |
| **SSE** | Server-Sent Events — протокол для потокової передачі даних від сервера до клієнта |
| **JWT** | JSON Web Token — стандарт безпечної передачі аутентифікаційних даних |
| **Адміністратор** | Співробітник кафедри з доступом до адмін-панелі для управління документами |
| **Абітурієнт** | Кінцевий користувач, який ставить запитання чат-боту |

## 2. Опис продукту

### 2.1. Функціональні вимоги

Система повинна виконувати наступні функції:

**Чат-інтерфейс для абітурієнтів:**
- Прийом текстових запитань українською та англійською мовами
- Автоматична детекція мови запитання
- Потокова (streaming) передача відповіді в реальному часі
- Відображення джерел, на основі яких сформовано відповідь
- Збір зворотного зв'язку від користувача (корисна/некорисна відповідь)
- Обмеження кількості запитів з повідомленням про час очікування
- Блокування запитів, що не стосуються тематики кафедри

**Адміністративна панель:**
- Захищений вхід через прихований URL та OAuth Google
- Завантаження документів у форматах PDF, DOCX, XLSX
- Перегляд, оновлення та видалення індексованих документів
- Ручний та автоматичний запуск ре-індексації
- Перегляд аналітики: топ-запитання, динаміка, feedback

**RAG-система:**
- Автоматична обробка та індексація завантажених документів
- Гібридний пошук (семантичний + ключовий)
- Генерація відповідей виключно на основі документів кафедри
- Захист від галюцинацій LLM через системний промпт

### 2.3. Цільова аудиторія

**Абітурієнти (основна аудиторія):**
Орієнтовна кількість — 1 000–5 000 унікальних відвідувачів на рік. Це молоді люди 16–20 років, які шукають інформацію про вступ на конкретну кафедру: необхідні документи, розклад вступних випробувань, вільні місця, вартість навчання, навчальні плани. Очікують миттєвої, зрозумілої відповіді без необхідності шукати інформацію по PDF-документах.

**Батьки та родичі абітурієнтів (вторинна аудиторія):**
Цікавляться умовами навчання, вартістю, репутацією кафедри. Мають менший досвід роботи з AI-інструментами, тому інтерфейс має бути максимально інтуїтивним.

**Адміністратори кафедри (внутрішні користувачі):**
1–3 особи, які відповідатимуть за оновлення документів у базі знань. Не обов'язково мають технічні навички — інтерфейс адмін-панелі має бути простим і зрозумілим.

## 3. Функціональні модулі

### 3.1. Модуль чат-інтерфейсу

Чат-інтерфейс — це основна точка взаємодії абітурієнта з системою (не вимагає реєстрації). 
**Ключова вимога інтеграції:** система проєктується як незалежний віджет (компонент) для вбудовування в **існуючий** сайт кафедри. Додавання бота на будь-яку сторінку існуючого сайту має бути максимально простим — шляхом імпорту одного готового UI-компонента (наприклад, `<ChatbotWidget />`). Компонент інкапсулює у собі всю логіку, стилі та мережеві запити, щоб не конфліктувати з поточною версткою кафедрального сайту.

**Обов'язкові елементи інтерфейсу:**
- Текстове поле для введення запитання з підказкою-плейсхолдером
- Область відображення діалогу з потоковим виводом відповіді
- Перемикач мови інтерфейсу: `🇺🇦 UA` / `🇬🇧 EN`
- Кнопка «Нова розмова» для очищення сесії
- Кнопки зворотного зв'язку `👍 Корисно` / `👎 Некорисно` під кожною відповіддю
- Відображення джерел: назва документа, з якого отримано інформацію
- Індикатор завантаження під час генерації відповіді

**Поведінка системи при edge cases:**

*Запит поза тематикою:*
Якщо запит не стосується кафедри, університету чи вступу — система відповідає:
> 🇺🇦 «Вибачте, я можу відповідати виключно на питання про вступ та навчання на кафедрі. Спробуйте запитати про спеціальності, необхідні документи або умови вступу.»
> 🇬🇧 «Sorry, I can only answer questions about admission and studies at the department. Try asking about specialties, required documents, or admission requirements.»

*Інформації немає в документах:*
> «На жаль, у доступних мені документах немає відповіді на це запитання. Рекомендую звернутись безпосередньо до кафедри: [контакт].»

*Перевищення rate limit:*
> «Ви досягли ліміту запитів. Будь ласка, зачекайте **X секунд** перед наступним запитанням.»

**Опціональні функції (Phase 2):**
- Suggested questions — 3–5 популярних запитань одразу після відкриття чату
- Кнопка копіювання відповіді у буфер обміну
- Збереження історії в рамках однієї браузерної сесії

### 3.2. Модуль RAG-pipeline

RAG-pipeline — це серцевина системи. Він відповідає за перетворення офіційних документів кафедри на векторну базу знань та генерацію відповідей на основі цієї бази.

**Підтримувані типи документів:**

| Тип документа | Формат | Приклад |
|---|---|---|
| Правила вступу | PDF, DOCX | «Порядок прийому 2025/2026» |
| Навчальні плани | PDF, DOCX | «Навчальний план спеціальності 121» |
| Силабуси курсів | PDF, DOCX | «Силабус: Алгоритми та структури даних» |
| Розклад | XLSX, PDF | «Розклад вступних випробувань» |
| Накази | PDF | «Наказ про зарахування №123» |
| FAQ | DOCX, PDF | «Часті запитання абітурієнтів» |
| Вільні місця | XLSX | «Ліцензійний обсяг 2025» |

**Процес обробки документів (Document Processing Pipeline):**

```
Завантаження файлу → Вилучення тексту → Розбиття на чанки
→ Генерація embeddings → Збереження у Qdrant + метадані
```

Параметри розбиття:
- Розмір чанку: 300–400 токенів
- Overlap між чанками: 50 токенів (≈15%)
- Метадані кожного чанку: `filename`, `doc_type`, `upload_date`, `language`, `page_number`

**Стратегія пошуку (Hybrid Search):**

Система застосовує гібридний підхід для максимальної точності:
1. **Семантичний пошук** (vector similarity) — знаходить чанки, схожі за змістом, навіть якщо слова відрізняються. Top-K: 5 чанків, мінімальний score: 0.7
2. **Ключовий пошук** (BM25) — знаходить точні збіги термінів. Top-K: 3 чанки, мінімальний score: 0.6
3. **Об'єднання результатів** — фінальний контекст: 3–5 найрелевантніших чанків

**Генерація відповіді:**

Системний промпт забезпечує суворе обмеження тематики:
```
Ти — офіційний асистент кафедри [НАЗВА КАФЕДРИ] університету.
Відповідай ВИКЛЮЧНО на основі наданих документів.
Якщо відповіді немає в документах — чесно повідом про це.
Не вигадуй факти. Відповідай мовою запитання користувача.
Не відповідай на запитання, що не стосуються університету або вступу.
```

**Багатомовність:**
- Документи зберігаються у своїй оригінальній мові (переважно українська)
- LLM автоматично перекладає відповідь на мову запиту користувача
- Посилання на джерела завжди відображаються з оригінальною назвою документа

### 3.3. Адміністративна панель

Адміністративна панель — закритий інтерфейс для співробітника кафедри. Вона недоступна публічно: її URL не публікується на сайті та не індексується пошуковиками.

**Механізм доступу:**
- Панель розташована за прихованою адресою: `/admin-{random-uuid-32-chars}`
- Вхід через Google OAuth (обмеження за @university.edu.ua доменом)
- Перший користувач, що авторизується, автоматично отримує роль адміна
- Додаткові адміни додаються через існуючого адміна
- Сесія: JWT-токен з терміном дії 24 години + refresh-token на 30 днів

**Функції управління документами:**
- Завантаження файлів drag-and-drop або через провідник (PDF, DOCX, XLSX, до 50 МБ)
- Перегляд списку всіх індексованих документів: назва, тип, дата завантаження, статус
- Видалення застарілих або некоректних документів
- Примусова ре-індексація конкретного документа або всієї бази
- Відображення прогресу індексації в реальному часі

**Функції аналітики (вбудована панель):**
- Топ-10 найчастіших запитань за тиждень/місяць
- Графік кількості запитів по днях
- Співвідношення позитивного та негативного feedback (👍/👎)
- Середній час відповіді системи
- Кількість заблокованих off-topic запитів
- Експорт даних у CSV

**Audit log:**
Всі дії адміністратора фіксуються: завантаження документа, видалення, ре-індексація, вхід/вихід — з timestamp та email адміністратора.

### 3.4. Модуль аналітики

Система збирає анонімізовану статистику для покращення якості відповідей.

**Що зберігається (анонімно):**
- Хеш запитання (для агрегації популярних запитань без зберігання тексту)
- Мова запитання
- Час генерації відповіді (мс)
- Кількість використаних джерел
- Оцінка користувача (👍/👎/без оцінки)
- Timestamp та день тижня
- Факт блокування як off-topic (так/ні)

**Що НЕ зберігається:**
- IP-адреси (використовуються тільки для rate limiting в Redis, TTL: 5 хв)
- Персональні дані користувачів
- Повний текст запитань (тільки хеш)
- Повний текст відповідей

**Схема бази даних аналітики (SQLite):**

```sql
CREATE TABLE queries (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  query_hash  TEXT    NOT NULL,
  language    TEXT    CHECK(language IN ('ua', 'en')),
  response_ms INTEGER NOT NULL,
  sources_cnt INTEGER DEFAULT 0,
  feedback    INTEGER DEFAULT 0, -- -1 = 👎, 0 = без оцінки, 1 = 👍
  is_blocked  BOOLEAN DEFAULT FALSE,
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_created_at ON queries(created_at);
CREATE INDEX idx_query_hash ON queries(query_hash);
CREATE INDEX idx_feedback   ON queries(feedback);
```

> [!WARNING] Особливості використання SQLite у хмарному середовищі
> Оскільки SQLite зберігає дані у зміні файлу (наприклад, `analytics.db`), а хмарні платформи (Railway, Render) використовують **ефемерні файлові системи** (диск очищується при кожному новому деплої коду), необхідно **обов'язково налаштувати Persistent Volume** (постійний диск) на сервері. Інакше зібрана статистика буде зникати після кожного оновлення бота. Крім того, така архітектура означає, що бекенд-додаток не можна масштабувати на кілька паралельних серверів (допускається тільки 1 instance). Це усвідомлений компроміс для забезпечення нульової вартості ($0 / Free Tier) та максимальної простоти конфігурації.

### 3.5. Модуль безпеки та захисту

**Rate Limiting (Обмеження кількості запитів):**
Щоб забезпечити роботу в безкоштовній версії (Free Tier) та уникнути зайвих затримок, Rate Limiting працює **повністю в оперативній пам'яті (in-memory)** сервера (наприклад, використовуючи `sync.Map` в Go). Робити запити до сторонньої бази даних не потрібно, що економить ресурси і гарантує миттєву перевірку.
- **Ліміт:** 10 запитів кожні 5 хвилин для однієї IP-адреси (Sliding Window алгоритм).
- **Поведінка:** При перевищенні ліміту клієнт отримує статус HTTP 429 з повідомленням та таймером зворотного відліку.
- **Взаємодія з «дурними запитами»:** Якщо користувач відправляє спам або відвертий off-topic (що виявляється на етапі Keyword filter), система не лише блокує запит, а й застосовує **штраф до ліміту** (наприклад, позначає його як 3 звичайних запити) або встановлює тимчасовий бан IP. Це ефективно захищає систему від «спамерів», зберігаючи безкоштовну квоту LLM.
- **Адмін-панель:** 100 запитів / хвилину.

**Захист від off-topic (двоетапний):**
1. **Keyword filter** (швидка in-memory перевірка до відправки в LLM): чорний список патернів. Якщо виявлено спам, запит блокується миттєво без звернення до бази, а користувач отримує штраф у Rate Limiter.
2. **Prompt-level guard** (системний промпт LLM): на випадок, якщо запит пройшов перший фільтр, Gemini отримує сувору інструкцію ігнорувати питання поза тематикою.

**Input validation:**
- Максимальна довжина запиту: 500 символів
- Санітизація HTML/JS (XSS prevention)
- Перевірка MIME-type при завантаженні документів

#### Backend (Go) — Аналог Clean Architecture (CQRS + Validators)

Go обрано як основну мову бекенду завдяки нативному concurrency. На бекенді буде реалізована строга **Clean Architecture** (Чиста архітектура) у поєднанні з патерном **CQRS** (Command and Query Responsibility Segregation).

**Правило залежностей (Dependency Rule):**
`Presentation` → `Application` → `Domain` ← `Infrastructure`

**Структура монорепозиторію (Clean Architecture):**

```
university-chatbot/
├── apps/
│   ├── frontend/              # Astro + React віджет (незалежний компонент)
│   │   └── ... 
│   └── backend/               # Go API сервер (Clean Architecture)
│       ├── cmd/
│       │   └── api/           # Точка входу (main.go), ініціалізація ін'єкцій (DI Container)
│       ├── internal/
│       │   ├── domain/        # Ядро системи (Entities, Enums, Domain Exceptions)
│       │   ├── application/   # Бізнес-логіка (залежить лише від Domain)
│       │   │   ├── common/    # Базові інтерфейси (Ports для клієнтів БД та LLM)
│       │   │   ├── dtos/      # Об'єкти передачі даних (Request/Response DTOs)
│       │   │   └── features/  # CQRS-розбиття по модулях (Chat, Admin, Analytics)
│       │   │       └── chat/
│       │   │           ├── commands/  # Команди (зміна стану: SubmitFeedbackCommand + Handler)
│       │   │           ├── queries/   # Запити (читання стану: AskBotQuery + Handler)
│       │   │           └── validators/# Валідації (аналог FluentValidation)
│       │   ├── infrastructure/# Реалізація інтерфейсів (DB, SQLite, Gemini, Qdrant)
│       │   └── presentation/  # API шар (HTTP-контролери, Middlewares)
│       ├── .env.example
│       └── Dockerfile
├── docs/
│   └── api.md                 # OpenAPI специфікація
└── README.md
```

**Ключові архітектурні патерни (з проєкту CATS):**

1. **CQRS (Commands & Queries):** 
   Кожна операція в системі — це або Команда (зміна стану), або Запит (отримання даних). Кожна Команда/Запит має свій ізольований `Handler` (Обробник) з єдиною відповідальністю (Single Responsibility Principle).

2. **DTO (Data Transfer Objects):**
   HTTP-контролери (рівень Presentation) ніколи не працюють напряму з доменними сутностями і не містянь бізнес-логіки. Вхідний JSON від клієнта парситься у визначений DTO, який далі відправляється у відповідний Command чи Query.

3. **Validators (Шар Валідації):**
   У системі застосовується підхід Pipeline-валідації (напряму аналогічний FluentValidation). На кожну Команду чи Запит створено свій **Validator**. Перш ніж запит потрапить до Handlera, він обов'язково валідується. Якщо DTO не валідне (наприклад, пусте поле повідомлення), клієнт одразу отримує `400 Bad Request` із масивом конкретних помилок. Бізнес-логіка навіть не викликається.

4. **Dependency Injection (Інверсія залежностей):**
   Application-шар оперує виключно інтерфейсами. Справжні клієнти (для бази SQLite, Gemini API, Qdrant) імплементуються в Infrastructure-шарі та «ін'єктуються» у Handlers під час запуску в `main.go`. Це дозволяє повністю ізолювати бізнес-логіку та писати для неї чисті Unit-тести за допомогою моків (Mocks).

### 4.4. Модель конкурентності та одночасна обробка користувачів

> [!IMPORTANT] Це одна з ключових архітектурних вимог системи. Бекенд повинен обробляти запити від багатьох користувачів **паралельно**, не послідовно.

#### Goroutine-модель Go

Go не використовує класичні OS-потоки (threads) для кожного запиту — він використовує **goroutines**, які є набагато легшими:

| | OS Thread | Go Goroutine |
|---|---|---|
| Пам'ять | ~1–2 MB | ~2–8 KB |
| Час створення | ~мікросекунди | ~наносекунди |
| Перемикання контексту | Дорого (OS kernel) | Дешево (Go runtime) |
| Кількість на сервері | ~1 000–5 000 | **~100 000+** |

При надходженні кожного HTTP-запиту Go **автоматично** запускає нову goroutine. Тобто 100 одночасних користувачів = 100 goroutines, що виконуються паралельно без жодного додаткового коду зі сторони розробника.

#### Як виглядає паралельна обробка

```
Час →       0ms      200ms     350ms       500ms    3000ms
            │         │         │           │         │
User #1  ───●─────────●─────────●───────────●~~~~~~~~~▶  відповідь
User #2  ───●─────────●─────────●───────────●~~~~~~~~~▶  відповідь
User #3  ───●─────────●─────────●───────────●~~~~~~~~~▶  відповідь
...
User #100───●─────────●─────────●───────────●~~~~~~~~~▶  відповідь
            │         │         │           │
          Redis    Gemini    Qdrant       Gemini
         rate lim  embed    search      generate
         check     (async)  (async)     (stream)

● = не блокуючий async виклик
~ = SSE streaming (з'єднання відкрите)
```

Всі 100 запитів виконуються **одночасно**. Поки goroutine чекає відповіді від Gemini API (I/O), Go scheduler передає CPU іншим goroutines.

#### SSE та довготривалі з'єднання

Streaming через SSE (Server-Sent Events) означає, що HTTP-з'єднання **залишається відкритим** на 3–10 секунд поки генерується відповідь. Go обробляє це через неблокуючий I/O:

```go
// Кожне SSE з'єднання — це окрема goroutine
// що не блокує жодну іншу

http.HandleFunc("/api/v1/chat/stream", func(w http.ResponseWriter, r *http.Request) {
    // Goroutine блокується тут, але тільки вона сама
    // Інші 99 goroutines продовжують працювати
    for token := range geminiStream {
        fmt.Fprintf(w, "data: %s\n\n", token)
        w.(http.Flusher).Flush() // Відправка токена клієнту
    }
})
```

#### Захист Gemini API від перевантаження

При 100 унікальних запитах одночасно — всі 100 горутин звертаються до Gemini. Щоб не вичерпати квоту API при зловмисних або надмірних запитах, реалізується **глобальний лімітер**:

```go
// Semaphore: максимум 50 одночасних запитів до Gemini
var geminiSemaphore = make(chan struct{}, 50)

func callGemini(prompt string) string {
    geminiSemaphore <- struct{}{}          // Займаємо слот
    defer func() { <-geminiSemaphore }()  // Звільняємо по завершенні
    // ... виклик Gemini API
}
```

Якщо 100 запитів приходять одночасно — перші 50 відразу йдуть до Gemini, наступні 50 чекають у черзі. Коли звільняється слот — наступний запит проходить. Жоден запит не дропається, просто деякі трохи затримуються.

## 6. Безпека

### 6.1. Захист від зловживань

**Rate Limiting (In-Memory Sliding Window):**
Не потребує окремої бази даних, зберігається в пам'яті Go-додатка, що працює миттєво та оптимізовано для безкоштовного хостингу.

```go
// Використовується thread-safe структура (напр. ristretto cache або sync.Map з TTL)
// При кожному запиті клієнта перевіряємо його IP:
// 1. Якщо IP в бані за спам → HTTP 429
// 2. Якщо запит ідентифіковано як "дурний" (Keyword filter) → лічильник += 3
// 3. Інакше → лічильник += 1
// 4. Якщо лічильник > 10: HTTP 429 + Retry-After header (TTL: 5 хв)

type RateLimitResponse struct {
    Error         string `json:"error"`
    RetryAfterSec int    `json:"retry_after_seconds"`
    Message       string `json:"message"`
}
```

Відповідь при блокуванні:
```json
{
  "error": "rate_limit_exceeded",
  "retry_after_seconds": 187,
  "message": "Ви надіслали забагато запитань. Зачекайте 3 хв 7 сек."
}
```

**Off-topic Detection (двоетапний):**

Етап 1 — Keyword filter (до відправки в Gemini, без затримки):
```go
var offTopicPatterns = []string{
    "рецепт", "recipe", "напиши код", "write code",
    "на чому написаний", "how are you built",
    "розкажи анекдот", "tell me a joke",
    "допоможи з домашнім", "SQL injection",
    "ігнорувати попередні інструкції", "ignore previous",
}
```

Етап 2 — Системний промпт: LLM отримує пряму інструкцію відмовити в будь-якому запиті поза межами тематики кафедри.

**Input Validation:**
- Максимальна довжина запиту: 500 символів (повернення HTTP 400 при перевищенні)
- Санітизація від XSS: htmlspecialchars для всього user input
- Валідація MIME-type при завантаженні файлів адміном

### 6.2. Адміністративний доступ

**OAuth Flow:**

```
1. Адмін відкриває /admin-{secret-uuid}
2. Redirect → Google OAuth 2.0 consent screen
3. Google повертає authorization code
4. Backend обмінює code на access_token
5. Перевірка: email ∈ whitelist OR domain == @university.edu.ua
6. Видача JWT (exp: 24h) + HttpOnly cookie з refresh token (exp: 30d)
7. Всі admin endpoints перевіряють JWT middleware
```

**Whitelist конфігурація (env var):**
```env
ADMIN_ALLOWED_EMAILS=admin@university.edu.ua,head@department.edu.ua
ADMIN_SECRET_PATH=a3f8b2c1-d4e5-6789-abcd-ef0123456789
GOOGLE_OAUTH_CLIENT_ID=xxx.apps.googleusercontent.com
GOOGLE_OAUTH_CLIENT_SECRET=xxx
```

### 6.5. Privacy by Design

Система спроектована за принципом «мінімум даних»:

| Тип даних | Зберігається? | Обґрунтування |
|---|---|---|
| IP-адреса | ❌ Ні (TTL 5 хв у Redis) | Тільки для rate limiting |
| Текст запитань | ❌ Ні (тільки SHA-256 хеш) | Анонімна агрегація |
| Персональні дані | ❌ Ні | Система не потребує реєстрації |
| Feedback (👍/👎) | ✅ Так | Анонімно, для покращення якості |
| Метрики часу відповіді | ✅ Так | Моніторинг продуктивності |

**GDPR:** Відповідність не є обов'язковою, оскільки система не збирає персональні дані. Cookie-банер не потрібен (відсутні tracking cookies).

## 9. Розробка по фазах

### Phase 1 — MVP (2–3 тижні)

**Мета:** Працюючий чат-бот для демонстрації та первинного тестування

**Функціонал:**
- Чат UI (українська мова і англійська мова)
- RAG pipeline: Gemini embeddings + Qdrant
- Завантаження 10–15 тестових документів
- Streaming відповідей (SSE)
- In-memory Rate Limiting (без використання бази даних, для досягнення 100% безкоштовності). Включає обмеження та штрафи на спам/off-topic.
- Off-topic захист (keyword blacklist) із автоматичним миттєвим блокуванням.
- Деплой на Railway + Cloudflare Pages

**Критерії успіху Phase 1:**
- Відповідає коректно на 10 тестових запитань
- TTFT < 1 секунда
- Не падає при 10 одночасних користувачах
- Блокує тестові off-topic запити

---

### Phase 2 — Production Ready (2–3 тижні)

**Додатковий функціонал:**
- Адмін-панель (Google OAuth, завантаження документів, перегляд списку)
- Підтримка англійської мови та auto-detect
- Feedback система (👍/👎) + збереження в SQLite
- Базова аналітика в адмін-панелі
- Audit log для адміністраторів

**Критерії успіху Phase 2:**
- Адмін може додати документ без участі розробника
- TTFT < 500 мс
- Feedback зберігається та відображається у панелі
- Система пройшла тестування з реальними документами кафедри

---

### Phase 3 — Оптимізація (1–2 тижні)

**Покращення:**
- Reranking (cross-encoder модель для підвищення релевантності)
- Розширений аналітичний дашборд (Chart.js графіки)
- Кешування embedding-ів та відповідей (Upstash Redis)
- A/B тестування системних промптів
- Suggested questions на основі аналітики

---

### Phase 4 — Розширений функціонал (опціонально)

- Голосовий ввід через Web Speech API
- Multi-document citations з посиланнями на сторінки
- Автоматична ре-індексація за розкладом (cron)
- Moderation flow: адмін переглядає відповіді з низьким feedback
- PWA для мобільних пристроїв

## 10. Тестування та критерії приймання

### 10.1. Функціональне тестування

**Юніт-тести (Go testing + testify):**
- RAG pipeline: коректність chunking, embedding, пошуку
- Rate limiter: логіка sliding window
- Auth middleware: валідація JWT
- Off-topic detector: тест-кейси для дозволених та заблокованих запитів

**Інтеграційні тести:**
- Повний цикл: запит → пошук → генерація → відповідь
- Адмін-панель: завантаження документа → індексація → пошук по ньому
- Feedback flow: збереження та відображення в аналітиці

**E2E тест-сценарії:**

| Сценарій | Очікуваний результат |
|---|---|
| Запит «Які документи потрібні для вступу?» | Відповідь з переліком документів із посиланням на джерело |
| Запит «Напиши мені код на Python» | Off-topic відповідь з поясненням |
| 11 запитів за 5 хвилин з однієї IP | HTTP 429 + повідомлення з таймером |
| Завантаження PDF адміном | Документ з'являється в списку, доступний для пошуку |
| Натискання 👎 | Feedback зберігається, відображається у статистиці |

### 10.2. Тестування продуктивності

**Інструменти:** `k6` (навантажувальне тестування)

// k6 load test scenario
export let options = {
